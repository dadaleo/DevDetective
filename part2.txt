1. 项目名称

中文名：开发者侦探
英文名：DevDetective
定位：AI 编程前置侦查工具
二期方向：开源工具 + Agent Skill + 网站体验版



2. 二期开发目标

一期已经完成基础测试：用户输入应用想法后，系统可以搜索 GitHub 上相似项目，并输出项目对比、最近更新时间、可复用评分和 Codex 开发提示词。

二期目标不是优先做收费 SaaS，而是将项目升级为：

可开源发布的开发者工具；
可本地运行的 GitHub 项目侦查工具；
可被 Codex / Cursor / claude/workbuddy其他 Agent 调用的 Skill；
可部署到网站的在线体验版；
可为 AI2Work 网站带来流量和开发者用户。
最终目标：

用户在开发新工具前，可以通过网页、本地命令行或 Codex Skill，先搜索 GitHub 上是否已有成熟开源项目，再决定 fork、参考，还是从零开发。



3. 二期产品策略

3.1 不优先做完整收费 SaaS

短期不做复杂付费系统，原因：

目标用户是开发者，容易倾向自己部署；
强行收费会降低使用和传播意愿；
付费系统会增加登录、额度、支付、客服和安全维护成本；
当前阶段更重要的是获取用户、GitHub Star、使用反馈和真实案例。
3.2 推荐策略

采用三层产品结构：

第一层：开源本地工具

第二层：Codex / Agent Skill,完成后再推进 Cursor / claude/workbuddy skill

第三层：网站在线体验版

具体定位：

DevDetective 开源免费，用户可fork自带 API Key 本地运行；

AI2Work 提供在线体验版和省事托管版。

4.2 用户自填 API Key

二期不要求用户登录，也不要求用户授权 GitHub OAuth。

用户本地运行时，自己填写：

OPENAI_API_KEY=

GITHUB_TOKEN=

或者兼容其他模型服务：

AI_PROVIDER=openai

OPENAI_API_KEY=



# 可选

ANTHROPIC_API_KEY=

DEEPSEEK_API_KEY=

OPENAI_BASE_URL=

说明：

GITHUB_TOKEN 用于提高 GitHub API 稳定性；
OPENAI_API_KEY 或兼容 API 用于需求拆解、README 总结、项目评分和 Codex Prompt 生成；
本地开源版不消耗项目方 API 成本；
用户数据默认保存在本地。

5. Agent Skill 版本

5.1 Skill 目标

二期需要新增一个独立目录：

devdetective-skill/

该 Skill 的目标是：

当用户让 Codex 或其他 Agent 开发一个新工具时，先不要直接写代码，而是先调用 DevDetective 的侦查流程，搜索 GitHub 上的相似项目，再输出 fork / 参考 / 从零开发建议。

5.3 SKILL.md 内容要求

SKILL.md 需要明确告诉 Codex：

当用户要求开发一个新的小工具、网页应用、桌面工具、浏览器插件、自动化脚本或 AI 工作流时，不要立即从零写代码。



你应该先执行 DevDetective 侦查流程：



1. 理解用户需求；

2. 生成 GitHub 搜索关键词；

3. 搜索相似开源项目；

4. 分析最近更新日期、维护状态、star、fork、license；

5. 判断是否适合 fork、参考架构、参考 UI 或仅借鉴局部功能；

6. 生成开发建议；

7. 再开始正式开发。

第二步：本地开源版可自填 API Key

7.3 GitHub OAuth 优先级

二期不建议做 GitHub OAuth。

原因：

当前主要搜索公开仓库；
用户无需授权也能完成核心流程；
OAuth 增加开发复杂度；
OAuth 主要用于私有仓库、一键 fork、star、issue、PR 等高级功能；
这些不是二期核心目标。
二期只需要：

本地版：用户填写自己的 GITHUB_TOKEN

网站版：服务端配置自己的 GITHUB_TOKEN，并做好限流

11. 维护状态判断规则

二期继续保留并强化最近更新时间逻辑。

需要展示：

创建日期 created_at

最近更新日期 updated_at

最近提交日期 pushed_at

距今多久未更新

维护状态标签

维护状态规则：

30 天内有提交：非常活跃

90 天内有提交：活跃

180 天内有提交：正常维护

365 天内有提交：低频维护

1–2 年未提交：维护风险

2 年以上未提交：疑似停更

已 archived：已归档，不建议直接使用

评分时优先使用 pushed_at，因为它更能代表代码是否仍在维护。

16. 错误处理

需要处理以下情况：

16.1 GitHub Token 未配置

提示：

当前未配置 GITHUB_TOKEN，可能会受到 GitHub API 请求限制。建议在 .env.local 中配置。

16.2 AI Key 未配置

提示：

当前未配置 AI API Key，无法进行需求拆解和项目总结。请在 .env.local 中配置 OPENAI_API_KEY 或兼容模型 API。

16.3 搜索结果为空

提示：

没有找到足够相似的项目。建议尝试：

1. 改用英文关键词；

2. 放宽技术栈限制；

3. 删除过窄的功能描述；

4. 改成搜索核心功能而不是完整产品。

16.4 GitHub API 限流

提示：

GitHub API 请求已达到限制。请稍后再试，或配置自己的 GITHUB_TOKEN。

2. 项目背景

当前越来越多开发者使用 Codex、Cursor、Claude Code 等 AI 编程工具快速开发小工具。但实际开发中存在一个明显问题：

很多需求在 GitHub 上已经有成熟开源项目，甚至维护多年，但用户往往没有先搜索和比较，而是直接让 AI 从零开发，导致消耗大量 token，最后得到一个半成品。

本项目希望解决这个问题：

在正式开发前，先根据用户的应用描述，自动搜索 GitHub 上最相近的开源项目，并判断哪些可以直接 fork，哪些适合参考，哪些不建议使用，最后生成可交给 Codex/Cursor 的二次开发提示词。

7. 最近更新日期规则

最近更新日期是本项目的关键判断依据之一。

需要同时展示：

项目创建日期；
最近更新日期；
最近提交日期；
距今多久未更新；
活跃状态标签。
7.1 活跃状态标签

根据 pushed_at 或 updated_at 判断：

30 天内有更新：非常活跃

90 天内有更新：活跃

180 天内有更新：正常维护

365 天内有更新：低频维护

1–2 年未更新：维护风险

2 年以上未更新：疑似停更

已 archived：已归档，不建议直接使用

7.2 展示示例

最近提交：2026-05-18

最近更新：2026-06-02

维护状态：活跃

创建时间：2021-03-12

维护年限：5 年

7.3 评分影响

最近更新日期进入综合评分。

建议规则：

30 天内更新：+20 分

90 天内更新：+16 分

180 天内更新：+12 分

365 天内更新：+8 分

1–2 年未更新：+3 分

2 年以上未更新：-10 分

archived：-30 分

注意：

updated_at 可能包括 issue、元数据等变化，pushed_at 更能代表代码层面是否仍在维护。因此展示时两个字段都保留，但评分时优先使用 pushed_at。
