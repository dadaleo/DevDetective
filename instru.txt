《开发者侦探 DevDetective》工程说明书

1. 项目名称

中文名：开发者侦探
英文名：DevDetective
备用名：开发伴侣 / BuildMate / GitHub 轮子侦探

推荐正式命名：

开发者侦探 DevDetective

推荐副标题：

写代码前，先查 GitHub 有没有成熟轮子。



2. 项目背景

当前越来越多开发者使用 Codex、Cursor、Claude Code 等 AI 编程工具快速开发小工具。但实际开发中存在一个明显问题：

很多需求在 GitHub 上已经有成熟开源项目，甚至维护多年，但用户往往没有先搜索和比较，而是直接让 AI 从零开发，导致消耗大量 token，最后得到一个半成品。

本项目希望解决这个问题：

在正式开发前，先根据用户的应用描述，自动搜索 GitHub 上最相近的开源项目，并判断哪些可以直接 fork，哪些适合参考，哪些不建议使用，最后生成可交给 Codex/Cursor 的二次开发提示词。



3. 产品定位

开发者侦探不是普通 GitHub 搜索工具，也不是代码生成工具。

它是一个：

AI 编程前置侦查工具

核心定位：

用户输入想做的工具，系统先帮他侦查 GitHub 上已有的相似开源项目，再给出复用建议和开发方案。

用户价值：

避免重复造轮子；
节省 AI 编程 token；
快速找到成熟项目；
判断项目是否仍在维护；
辅助生成 Codex 二次开发提示词；
帮助非资深开发者理解哪些开源项目值得参考。


4. 目标用户

4.1 核心用户

独立开发者；
使用 Codex / Cursor / Claude Code 的 AI 编程用户；
需要快速做 MVP 的创业者；
小团队产品经理；
技术能力中等但不熟悉 GitHub 高级搜索的人；
想找开源替代方案的开发者。
4.2 典型场景

用户输入：

我想做一个本地运行的视频号转文字归档工具，可以每天把视频内容转成文字，按主题分类保存。

系统输出：

需求理解；
推荐搜索关键词；
相似 GitHub 项目 Top 10；
每个项目的 star、fork、license、最近更新日期、最近提交日期；
可复用评分；
是否建议 fork；
适合参考的模块；
给 Codex 的二次开发提示词。


5. MVP 目标

第一版只做一个核心闭环：

输入需求 → 搜索 GitHub → 项目评分 → 对比总结 → 输出 Codex 开发提示词

不做复杂社区、登录系统、多人协作、自动 fork、自动部署。

6.4 GitHub 搜索策略生成

系统根据需求生成多组 GitHub 搜索关键词。

示例：

prompt manager local app language:TypeScript

markdown json import tool language:Python

local knowledge base electron

prompt manager stars:>50 pushed:>2024-01-01

搜索策略要求：

同时生成中文和英文关键词；
英文关键词优先，因为 GitHub 项目主要以英文命名；
至少生成 5 组搜索词；
每组搜索词尽量聚焦一个功能方向；
不要只搜索宽泛词，例如 “tool”“app”；
对最近更新项目增加 pushed 条件；
对高质量项目增加 stars 条件；
对轻量项目可以降低 star 门槛，避免漏掉小而精准的项目。

7. 最近更新日期规则

最近更新日期是本项目的关键判断依据之一。

需要同时展示：

项目创建日期；
最近更新日期；
最近提交日期；
距今多久未更新；
活跃状态标签。
7.1 活跃状态标签

根据 pushed_at 或 updated_at 判断：

30 天内有更新：非常活跃

90 天内有更新：活跃

180 天内有更新：正常维护

365 天内有更新：低频维护

1–2 年未更新：维护风险

2 年以上未更新：疑似停更

已 archived：已归档，不建议直接使用

7.2 展示示例

最近提交：2026-05-18

最近更新：2026-06-02

维护状态：活跃

创建时间：2021-03-12

维护年限：5 年

7.3 评分影响

最近更新日期进入综合评分。

建议规则：

30 天内更新：+20 分

90 天内更新：+16 分

180 天内更新：+12 分

365 天内更新：+8 分

1–2 年未更新：+3 分

2 年以上未更新：-10 分

archived：-30 分

注意：

updated_at 可能包括 issue、元数据等变化，pushed_at 更能代表代码层面是否仍在维护。因此展示时两个字段都保留，但评分时优先使用 pushed_at。



8. 项目评分体系

每个搜索结果生成一个“可复用评分”。

总分 100 分。

建议权重：

需求相似度：35 分

维护活跃度：20 分

项目成熟度：15 分

License 友好度：10 分

文档完整度：10 分

代码可改造性：10 分

8.1 需求相似度

AI 根据 README、项目描述、topics、项目名判断。

评分维度：

是否解决同类问题；
是否包含用户需求中的核心功能；
技术形态是否匹配；
是否适合作为底座；
是否只是部分相关。
输出标签：

高度相似

中度相似

部分可参考

仅概念相关

不相关

8.2 维护活跃度

根据：

最近提交日期；
最近更新日期；
是否 archived；
release 是否近期更新；
issue 是否长期无人处理。
第一版只做基础字段：

pushed_at

updated_at

archived

open_issues_count

第二版再加入 release、PR、commit history。

8.3 项目成熟度

参考：

star 数；
fork 数；
创建时间；
是否维护多年；
是否有较完整 README；
是否有 demo；
是否有 release。
注意：star 不是唯一指标。小项目 star 不高，但如果功能高度匹配，也应进入推荐列表。

8.4 License 友好度

展示 license。

优先级：

MIT / Apache-2.0 / BSD：友好

GPL / AGPL：需谨慎

未声明 license：高风险

系统不能给法律结论，只能提示：

该项目未声明 license，商业化或二次分发前需自行确认授权风险。

8.5 文档完整度

根据 README 判断：

是否有安装说明；
是否有截图；
是否有 usage；
是否有配置说明；
是否有贡献说明；
是否有 Demo 链接。
8.6 代码可改造性

第一版可以由 AI 根据 README 和项目结构粗略判断。

判断项：

技术栈是否常见；
是否过度复杂；
是否前后端耦合严重；
是否适合本地部署；
是否依赖大量第三方服务；
是否适合交给 Codex 二次开发。

15. 评分逻辑伪代码

function calculateRepoScore(repo, aiSimilarityScore) {

  const similarity = aiSimilarityScore * 0.35;

  const activity = calculateActivityScore(repo.pushed_at, repo.updated_at, repo.archived) * 0.20;

  const maturity = calculateMaturityScore(repo.stars, repo.forks, repo.created_at) * 0.15;

  const license = calculateLicenseScore(repo.license) * 0.10;

  const docs = calculateDocsScore(repo.readme_text) * 0.10;

  const modifiability = calculateModifiabilityScore(repo) * 0.10;



  return Math.round(similarity + activity + maturity + license + docs + modifiability);

}

15.1 活跃度评分

function calculateActivityScore(pushedAt, updatedAt, archived) {

  if (archived) return 0;



  const date = pushedAt || updatedAt;

  const days = daysSince(date);



  if (days <= 30) return 100;

  if (days <= 90) return 80;

  if (days <= 180) return 65;

  if (days <= 365) return 45;

  if (days <= 730) return 20;



  return 5;

}

15.2 维护状态标签

function getMaintenanceLabel(pushedAt, updatedAt, archived) {

  if (archived) return "已归档";



  const date = pushedAt || updatedAt;

  const days = daysSince(date);



  if (days <= 30) return "非常活跃";

  if (days <= 90) return "活跃";

  if (days <= 180) return "正常维护";

  if (days <= 365) return "低频维护";

  if (days <= 730) return "维护风险";



  return "疑似停更";

}

17. AI 提示词设计

17.1 需求拆解 Prompt

你是一个开发需求分析助手。

请根据用户输入，提取这个应用想法的核心功能、可能技术栈、GitHub 搜索关键词。



要求：

1. 同时输出中文关键词和英文关键词；

2. 英文关键词要适合 GitHub 搜索；

3. 不要生成太宽泛的词；

4. 至少生成 5 组搜索 query；

5. 输出 JSON。

17.2 项目相似度判断 Prompt

你是一个开源项目分析助手。

请根据用户需求和 GitHub 项目信息，判断该项目是否适合作为开发参考。



你需要判断：

1. 功能是否相似；

2. 技术栈是否合适；

3. 是否可以直接 fork；

4. 是否只适合参考局部功能；

5. 是否不建议使用；

6. 简要说明原因。



请输出：

- similarity_score: 0-100

- reuse_level: 建议 fork / 参考架构 / 参考 UI / 借鉴局部功能 / 仅供调研 / 不建议使用

- reason: 简短原因

17.3 Codex Prompt 生成 Prompt

你是一个 AI 编程任务规划助手。

用户已经找到若干相似 GitHub 项目，请你根据用户需求和这些项目，生成一段可以交给 Codex/Cursor 的二次开发提示词。



要求：

1. 明确推荐基础项目；

2. 说明保留什么；

3. 说明新增什么；

4. 说明避免什么；

5. 分阶段给出开发步骤；

6. 强调先阅读 README 和项目结构；

7. 输出中文。

23. 风险与注意事项

23.1 GitHub API 限制

必须做：

GitHub Token 配置；
请求缓存；
搜索去重；
README 限量抓取；
错误提示；
请求失败 fallback。
23.2 AI 判断不稳定

解决方式：

评分规则中保留硬指标；
不完全依赖 AI；
star、fork、pushed_at、license 等字段先做基础评分；
AI 只负责语义相似度和总结；
输出中标注“建议仅供参考”。
23.3 License 风险

系统只做提示，不做法律判断。

提示文案：

该项目 license 可能影响二次分发或商业使用，请在正式使用前自行确认授权条款。

23.4 搜索结果不准确

解决方式：

展示搜索关键词；
支持重新生成关键词；
支持用户手动修改关键词；
支持排除不相关项目；
支持选择更准确的技术栈。
